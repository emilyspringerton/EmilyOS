Done. “Game” becomes a first-class verb with the same mechanical dignity as Pause/Resume/Withdraw/Exit.

1) New verb
VERB_GAME
Meaning: enter/exit the game space — a local, offline, bounded execution domain that never requires legacy compatibility.

Key rule: GAME is not “a program.” GAME is a posture-preserving space.

2) State machine update
States
STATE_STILL (default)
STATE_OPEN (legacy/work space is conceptually available)
STATE_GAME (game space active)
STATE_PAUSED
STATE_EXITED
Transitions (normative)
From STATE_STILL

ENTER → STATE_OPEN
GAME → STATE_GAME
EXIT → STATE_EXITED
From STATE_OPEN

GAME → STATE_GAME
PAUSE → STATE_PAUSED
EXIT → STATE_EXITED
From STATE_GAME

GAME → STATE_STILL (toggle behavior: game is a room you leave)
PAUSE → STATE_PAUSED (optional; see below)
EXIT → STATE_EXITED
From STATE_PAUSED

RESUME → return_to_previous_state (must remember prior: OPEN or GAME)
EXIT → STATE_EXITED
Important: GAME is toggleable and must be instant (no loading spinner behavior).

3) Mechanical contract for GAME
G0 — Offline by construction
While STATE_GAME is active:

cap.net is treated as OFF (hard-off), regardless of prior grants.
No network devices in the game domain.
No background update checks.
No “phone home,” ever.
G1 — Stillness-first rendering
The game may render frames only when it must:

If the game is paused or idle, it must fall back to WFI/HLT and redraw only on input/state change.
If the game is active, it may run a frame loop — but no “smoothing as aliveness.” (No hidden queues, no vsync chasing policy.)
G2 — Deterministic resource budget
GAME must be bounded:

fixed CPU quota (or tick budget)
fixed memory cap
fixed storage cap (prefer tmpfs + explicit save/export)
G3 — No bridge leakage
While in GAME:

Legacy domains are not callable by default.
No host filesystem browsing.
The only allowed boundary crossing is explicit EXPORT (if you permit it).
4) Relationship to Siege and Mercy
In SIEGE
GAME is allowed and becomes a safe room:

all legacy domains stopped unless pinned
network hard-off
game runs offline
Rationale: Siege is “hostile outside.” Game is “protected inside.”

In MERCY
GAME is preferred:

entering GAME implicitly enforces Mercy’s quiet clamp
game cannot increase system “demand” (tight budgets)
Rationale: Mercy is “operator relief.” Game is relief you can enter deliberately.

5) Implementation mapping
You have two clean ways to implement GAME as first-class without undermining the architecture.

Option A (recommended): GAME is a separate “native domain”
Runs as a host-managed process (or its own container) with:

no net
no host mounts
strict resource caps
UI and input route through EmilyOS, not the game.

API mapping

verb:GAME → game.start() / game.stop()
Minimal game API:

game.start(profile_id)
game.stop(reason)
game.snapshot(tag) (optional)
game.export(artifact_id | path_spec) (optional)
Option B: GAME is a pinned “domain template”
Treat game as a special LXC domain with a locked template:

template: game-locked
rootfs immutable
overlay optional
net always off
exec only allows the game binary
This keeps everything unified under domain.*, but still makes GAME first-class at the verb layer.

6) Policy changes to the legacy domain contract
Add posture gate:

Global rule
If STATE_GAME is active, then:

domain.exec is DENIED unless:

it is status, stop, or export
domain.start is DENIED unless pinned and explicitly allowed

This prevents the “I’m gaming but the legacy world is still tugging” failure mode.

7) UX rule (to keep the vibe correct)
When GAME is invoked:

no confirmation prompts
no menus unless you explicitly asked for them
the transition must be one frame: stillness → game, game → stillness
Optional minimal indicator (allowed):

a single static glyph change (e.g. emily → emily▮) No animation.
